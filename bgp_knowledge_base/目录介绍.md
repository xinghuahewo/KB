# BGP 知识库目录介绍

## 项目概览

```
bgp_knowledge_base/
├── config/          📋 配置与分类体系（YAML）
├── raw/             📦 不可变原始资料（PDF/TXT/HTML/YAML）
├── parsed/          📄 解析后的结构化文本（JSON）
├── cleaned/         📝 清洗后的可读文本（Markdown）
├── chunks/          🧩 知识片段（JSONL）
├── entities/        🏷️ 结构化领域实体（JSONL）
├── relationships/   🔗 实体关系图谱（JSONL）
├── schemas/         ✅ 数据格式约束（JSON Schema）
├── inventory/       📊 来源登记表（CSV）
├── datasets/        📈 规则化派生数据集（JSONL + CSV）
├── published/       🚀 可交付发布包（SQLite + JSONL + JSON）
├── reports/         📋 报告（Markdown）
├── review_inputs/   ✍️ 人工复核输入（CSV + 模板）
├── scripts/         🐍 流水线脚本（Python）
└── README.md        📖 项目总览
```

数据流向：

```
raw → parsed → cleaned → chunks → entities → relationships → published
                    └──────────── datasets ─────────────┘
                    └──────────── reports ──────────────┘
```

## 文档归并入口

| 入口 | 先解决的问题 |
| --- | --- |
| `README.md` | 这个知识库是什么、怎么跑、人工复核怎么进入。 |
| `docs/README.md` | 项目方案、治理文档和目录导航应按什么顺序阅读。 |

| `published/README.md` | 可交付知识库包如何读取和查询。 |

## config/ — 配置与分类体系（5 个 YAML 文件）

定义知识库的元规则，是流水线的配置文件。

| 文件 | 作用 | 何时用 |
|---|---|---|
| `source_types.yaml` | 来源类型枚举（standard / paper / data_doc / case_report / blog / manual_note） | 登记新来源时参考 |
| `topic_taxonomy.yaml` | BGP 主题分类体系（BGP基础 → 路径与属性 → BGP安全 → ...） | 给 chunk 或实体打标签 |
| `entity_types.yaml` | 9 种实体类型定义（BGPConcept / AnomalyType / DataSource / ...） | 创建新实体时参考 |
| `quality_rules.yaml` | 质量检查规则（每个实体类型必须有哪些字段、9 项全局检查） | 了解质量门禁 |
| `storage_formats.yaml` | 各数据层存储格式约定 | 了解数据规范 |

**如何检查**：打开任意 YAML 文件阅读即可，全是纯文本配置。

## raw/ — 原始资料层（53 个文件，不可修改）

所有外部资料的原始形式，按来源类型分目录。这一层**只读不写**，保证追溯。

| 子目录 | 数量 | 内容 |
|---|---|---|
| `standards/` | 11 个 | RFC 文本（4271, 6480, 6811, 7908, 8205, 8210, 9234, 2622, 3912, 9082, 9083） |
| `data_docs/` | 14 个 | 数据源文档（RouteViews API、RIPE RIS、CAIDA、RIPEstat、PeeringDB、MANRS 等） |
| `papers/` | 13 个 | 学术论文 PDF（BEAR、BEAM、BGPShield、ARTEMIS、BGP2Vec、ASwatch 等） |
| `cases/` | 13 个 | 案例报告 HTML/PDF（YouTube 劫持、Facebook 宕机、Cloudflare 事件等） |
| `tools_docs/` | 2 个 | 工具文档（BGPStream） |
| `notes/` | 0 个 | 人工笔记（预留，当前为空） |

**如何检查**：直接打开文件确认内容完整性和格式正常。比如：

```bash
head -50 raw/standards/rfc4271.txt
open raw/papers/bear_2025.pdf
```

## parsed/ — 解析文本层（53 个 JSON 文件）

从 raw 确定性解析出的结构化文本，保留了章节层级、页码、图表标题等结构信息。

**关键字段**：`doc_id`, `title`, `sections[].heading`, `sections[].content`, `sections[].page_start/end`, `source_path`

**如何检查**：

```bash
python3 -m json.tool parsed/papers/bear_2025.json | head -80
python3 -m json.tool parsed/standards/rfc4271.json | head -50
```

---

## cleaned/ — 清洗文本层（54 个 Markdown 文件）

从 parsed 清洗后的纯文本，去除了页眉页脚、修复了断词、统一了写法，适合直接阅读或做 chunk 切分。

**如何检查**：

```bash
head -100 cleaned/papers/bear_2025.md
head -100 cleaned/standards/rfc4271.md
```

## chunks/ — 知识片段层（2,037 条 JSONL + 1 个 seed 文件）

从 cleaned 文本按固定规则切分的知识片段，每条 chunk 都有明确的来源追溯。

| 文件 | 条数 | 内容 |
|---|---|---|
| `bgp_chunks.jsonl` | 890 | 通用 BGP 概念、数据源说明、工具文档的 chunk |
| `standard_chunks.jsonl` | 554 | RFC 标准文档按 section 切分的 chunk |
| `paper_chunks.jsonl` | 504 | 论文方法按 section 切分的 chunk |
| `case_chunks.jsonl` | 89 | 案例报告按事件要素切分的 chunk |
| `seeds/context_chunks.jsonl` | — | 从 context.md 切出的 seed chunk（提供项目范围和知识边界） |

**每条 chunk 的结构**：

```json
{
  "chunk_id": "bear_2025_sec3_2_001",
  "doc_id": "bear_2025",
  "chunk_type": "paper_method",
  "topics": ["BGPStream", "RIB", "AS_PATH"],
  "content": "...",
  "source_ref": "BEAR.pdf:p4:III-B",
  ...
}
```

**如何检查**：

```bash
# 统计各类 chunk 数量
wc -l chunks/*.jsonl

# 抽查一条
head -1 chunks/bgp_chunks.jsonl | python3 -m json.tool

# 全文搜索（通过 SQLite）
python3 scripts/query_knowledge_base.py search-chunks '"route leak"' --limit 5
```

| 文件 | 条数 | 审批状态 | 内容 |
|---|---|---|---|
| `bgp_concepts.jsonl` | 31 | ✅ 31 approved | BGP 基础概念（AS, Prefix, AS_PATH, RPKI, ...） |
| `data_fields.jsonl` | 32 | ✅ 32 approved | BGP 数据字段语义（prefix, as_path, roa_status, ...） |
| `routing_mechanisms.jsonl` | 12 | ✅ 12 approved | 路由机制（Path-vector, Valley-free, ROV, ASPA, BGPsec, ...） |
| `data_sources.jsonl` | 9 | ✅ 9 approved | 外部数据源说明（RouteViews, RIPE RIS, BGPStream, CAIDA, ...） |
| `anomaly_types.jsonl` | 8 | ✅ 8 approved | 异常类型定义（Hijack, Route Leak, MOAS, Outage, ...） |
| `evidence_templates.jsonl` | 8 | ✅ 8 approved | 每类异常需要的证据模板 |
| `cases.jsonl` | 5 | ✅ 1 approved + 4 needs_source | 历史 BGP 事件案例 |
| `false_positive_patterns.jsonl` | 4 | ✅ 4 approved | 误报与边界模式 |

## relationships/ — 关系组织层（106 条 JSONL）

描述实体之间的语义关系（如 `requires_evidence`、`supports`、`belongs_to`、`provides` 等）。

**关系类型举例**：

- `AnomalyType → requires_evidence → EvidenceTemplate`
- `BGPConcept → supports → AnomalyType`
- `DataSource → provides → DataField`
- `FalsePositivePattern → guards_against_overclaiming → AnomalyType`

**如何检查**：

```bash
# 查某个实体的所有关系邻居
python3 scripts/query_knowledge_base.py neighbors concept_as_path
python3 scripts/query_knowledge_base.py neighbors anomaly_route_leak

# 直接看文件
head -5 relationships/relationships.jsonl | python3 -m json.tool
```

## schemas/ — JSON Schema 约束层（35 个文件）

```bash
# 看某个实体类型的必填字段
```

## datasets/ — 规则化派生数据集（44 个文件）

流水线从 entities / chunks / relationships 机械派生的中间数据集，用于支持人工复核、质量检查和发布。每个数据集同时提供 `.jsonl`（机器可读）和 `.csv`（人类可在表格工具中打开）。

| 数据集 | 用途 |
|---|---|
| `entity_review_queue.*` | 待人工复核实体列表 |
| `entity_review_packets.*` | 每个实体对应的来源证据包 |
| `entity_source_evidence.*` | 实体→来源→chunk 的证据映射（246 条） |
| `human_review_workbook.*` | 人工复核工作簿（112 条） |
| `human_review_evidence_extracts.*` | 每个实体对应的 top-6 chunk 摘录（672 条） |
| `human_review_field_checklist.*` | 逐字段核验清单（834 项） |
| `human_review_session_queue.*` | 12 个复核批次的分组 |
| `human_review_progress.*` | 复核进度统计 |
| `human_review_source_matrix.*` | 按来源交叉引用实体和复核状态 |
| `human_review_decision_audit.*` | 复核决策审计结果 |
| `glossary.*` | 从实体机械派生的术语表（112 条） |
| `case_observations.*` | 从案例 chunk 正则抽取的观察值（148 条） |
| `next_action_queue.*` | 下一步行动队列（114 条） |
| `artifact_manifest.*` | 制品清单（含文件大小和 SHA-256） |
| `source_processing_status.*` | 每个来源的处理状态 |
| `source_gap_queue.*` | 未完成来源缺口队列 |

**如何检查**：

```bash
# CSV 可直接在 Excel/Numbers 中打开
open datasets/human_review_workbook.csv
open datasets/entity_review_queue.csv

# JSONL 可抽样查看

```

## scripts/ — 流水线脚本（45 个 Python 文件）

所有数据处理、质量检查和发布脚本。统一入口：

```bash
python3 scripts/run_pipeline.py        # 一键运行全部确定性流水线
python3 scripts/query_knowledge_base.py # 查询入口
```

**核心脚本分类**：

| 类别 | 脚本 | 作用 |
|---|---|---|
| 解析 | `parse_documents.py` | 从 raw 解析为 parsed JSON |
| 清洗 | `clean_text.py` | 从 parsed 清洗为 cleaned Markdown |
| 切分 | `build_chunks.py` | 从 cleaned 切分为 chunks JSONL |
| 实体 | `extract_entities.py` | 从 chunks 抽取实体（当前为人工整理入口） |
| 关系 | `build_relationships.py` | 构造实体关系图 |
| 质量 | `quality_check.py` | 全面质量校验 |
| 案例 | `extract_case_observations.py` | 从案例 chunk 机械抽取观察值 |
| 人工复核 | `build_human_review_*.py`（20 个） | 生成工作簿/模板/校验/审计/进度/摘录/会话队列/字段清单/来源矩阵/任务板/交接清单/指南 |
| 应用 | `apply_human_review_decisions.py` | 将人工复核决策写入实体文件 |
| 发布 | `build_published_knowledge_base.py` | 生成 published/ 发布包 |
| 发布 | `build_sqlite_knowledge_base.py` | 生成 SQLite 数据库 |
| 发布 | `build_published_integrity_report.py` | 校验 SQLite vs manifest 一致性 |
| 查询 | `query_knowledge_base.py` | CLI 查询入口（stats / entity / neighbors / evidence / search-chunks / glossary 等） |
| 报告 | `build_*.py`（其余） | 生成各类 Markdown 报告 |

```bash
python3 scripts/query_knowledge_base.py neighbors anomaly_route_leak
```

确认输出显示了 requires_evidence、supports 等关系，上下游实体都能找到。

```bash
sqlite3 published/bgp_knowledge_base.sqlite
```
