# BGP 对话前端设计调整实施计划 v1

## 1. 目标

在不引入登录、账号、云端会话和后端 API 破坏性变更的前提下，完成 BGP 对话前端的布局、证据体验、会话状态、输入交互、可访问性和长对话性能调整。

本轮的核心目标是：

1. 消除三栏页面的整页滚动、双滚动和断点跳变问题。
2. 让手机和平板始终以对话为主，历史与证据按需打开。
3. 让每条回答保留自己的检索状态、引用和上下文，避免多轮对话串证据。
4. 正确区分“文档来源数”和“证据片段数”，避免错误的可信度提示。
5. 改善回答排版、证据可读性、请求取消、错误恢复和键盘/触控体验。
6. 建立能覆盖滚动、响应式和证据状态的回归验证。

## 2. 明确边界

- 不实现登录、注册、用户资料、权限或云端同步。
- 不部署到远端服务器；本轮只完成本地实现与验证。
- 不改变现有 FastAPI 问答接口的请求和响应契约。
- 不删除或覆盖工作区中与 `chat_frontend` 无关的现有修改。
- 不把 chunk 哈希、内部文件路径和原始检索 JSON 作为默认用户界面的主要内容。

## 3. 视觉与交互方向

采用“技术编辑式网络运维工作台”方向：

- 暖白纸张背景、深墨色正文、森林绿用于证据与健康状态，警告和错误使用克制的琥珀色与红色。
- 中文宋体或合适的系统衬线字体用于标题，无衬线字体用于正文，等宽字体只用于确有必要的协议标识。
- 减少卡片套卡片和重复边框，优先使用留白、字号、背景层次和分组标题。
- 桌面正文保持约 720～820px 的舒适阅读宽度。
- 英文大写界面标签统一为中文；RFC、BGP、RPKI 等标识保持原样。
- 动效只用于抽屉和状态切换，支持 `prefers-reduced-motion`。

## 4. 实施任务

### 任务 1：建立基线与回归测试

涉及文件：

- `chat_frontend/package.json`
- `chat_frontend/vitest.config.ts`
- `chat_frontend/tests/`

步骤：

1. 在隔离 worktree 中运行 `yarn test` 与 `yarn build`，记录基线结果。
2. 仅在确有必要时增加轻量组件测试依赖；优先把来源归一化、会话迁移和展示模型提取为可在 Node 环境测试的纯函数。
3. 按测试驱动方式先增加失败测试，至少覆盖：
   - 同一文档的多个 `source_ref` 片段只计为 1 个来源。
   - 证据片段数与文档来源数分别计算。
   - 每条 assistant 消息能够保存独立证据。
   - 旧版 localStorage 数据能够迁移或安全加载。
   - 用户取消请求时返回“已停止”而不是“超时”。
4. 观察测试因缺少实现而按预期失败后，再进入生产代码修改。

验收条件：新增测试经历明确的红—绿过程；原有测试保持通过。

### 任务 2：修复视口与响应式布局

涉及文件：

- `chat_frontend/components/chat/chat-shell.tsx`
- `chat_frontend/components/layout/app-sidebar.tsx`
- `chat_frontend/components/layout/top-status-bar.tsx`
- `chat_frontend/components/chat/citation-panel.tsx`
- `chat_frontend/app/globals.css`

步骤：

1. 根布局使用动态视口高度和受控溢出；中栏使用 `min-h-0`，仅消息列表滚动。
2. 顶部状态栏和输入区保持在中栏可见范围内，不随回答正文离开视口。
3. 桌面宽度达到 `xl` 时使用辅助栏、对话主体和按需证据栏；没有证据时不永久占用右栏空间。
4. 低于 `xl` 时只显示对话主体；示例/本地会话和证据分别通过顶部按钮打开抽屉。
5. 抽屉支持遮罩点击关闭、Escape 关闭、焦点可见、`overscroll-behavior: contain` 和安全区内边距。
6. 修复长文本、来源标签和 flex 子项产生的横向溢出。

验收条件：

- 1280×720 长回答下，页面根节点不产生纵向滚动；消息区和证据区各自在限定区域滚动。
- 1024px 与 1023px 不再出现三栏到超长纵向页面的突变。
- 390px 下不出现横向滚动，首屏直接展示对话主体。

### 任务 3：重建回答轮次与证据状态

涉及文件：

- `chat_frontend/lib/chat-types.ts`
- `chat_frontend/lib/storage.ts`
- `chat_frontend/components/chat/chat-shell.tsx`
- `chat_frontend/components/chat/message-list.tsx`
- 新增的证据展示辅助模块

步骤：

1. 为 assistant 消息增加回答状态、引用、检索摘要和上下文包字段，保持用户消息轻量。
2. 新问题提交时立即切换到“本轮检索中”的空证据状态，不继续展示上一轮证据。
3. 回答完成后把证据写入对应 assistant 消息；选择历史回答时能够重新查看该轮证据。
4. localStorage 增加 schema 版本，迁移旧的全局 `citations/retrieval` 数据，并保留原会话 ID。
5. 存储写入改为回答完成后或防抖写入，避免每个进度阶段同步序列化整段会话。

验收条件：连续询问 2 个问题后，两条回答分别保留自己的证据；刷新页面后仍可查看。

### 任务 4：修正来源统计并优化证据卡

涉及文件：

- `chat_frontend/components/chat/citation-panel.tsx`
- `chat_frontend/components/chat/retrieval-status.tsx`
- 新增的来源归一化与分组模块

步骤：

1. 以文档 ID 或去除片段定位后的 canonical source key 统计来源数。
2. 明确分别显示“文档”和“证据片段”，不把片段数称作来源数。
3. 按文档分组引用；默认显示文档标题、RFC/案例名称、章节、摘录和片段数量。
4. chunk 哈希、内部路径、原始 score 和原始检索 JSON 移入折叠的技术详情；生产默认界面不显示调试 JSON。
5. 回答卡提供“查看 N 个文档 / M 条证据”动作，点击后打开并定位到该轮证据。

验收条件：同一 RFC 的多个片段只产生一个来源分组；所有长标识均能换行或截断。

### 任务 5：回答排版与输入交互

涉及文件：

- `chat_frontend/components/chat/message-list.tsx`
- `chat_frontend/components/chat/message-composer.tsx`
- `chat_frontend/lib/rag-stream.ts`

步骤：

1. 使用安全的 Markdown 渲染方式支持段落、列表、强调、代码和链接；不启用原始 HTML。
2. 为回答正文设置稳定的排版尺度和任意长 token 换行策略。
3. 输入框自动增高到上限，并显示“Enter 发送，Shift+Enter 换行”。
4. 将流式请求的内部超时信号与外部取消信号组合，提供真正的“停止生成”。
5. 区分用户取消、请求超时、服务不可用和证据不足；分别提供重试或改写问题的下一步操作。
6. 复制、状态更新和错误提示使用可访问的即时反馈。

验收条件：Markdown 不再以原始星号显示；停止按钮能取消请求；错误状态不会同时显示“知识库就绪”。

### 任务 6：可访问性与视觉收敛

涉及文件：

- `chat_frontend/app/layout.tsx`
- `chat_frontend/app/globals.css`
- `chat_frontend/components/**/*.tsx`

步骤：

1. 为图标按钮、输入框、两个 complementary landmark 和异步状态补充可访问名称或 live region。
2. 统一按钮、链接、summary 和抽屉控件的 `focus-visible` 样式。
3. 触控目标尽量达到 44×44px，并设置明确的 tap highlight 与 touch action。
4. 清空会话提供确认或短时撤销，不再立即无提示删除。
5. 统一中文界面文案、数字格式和省略号。
6. 删除重复边框和无价值的静态统计，保留清晰的视觉层级。

验收条件：键盘能够完成提问、打开/关闭抽屉、查看证据、复制和清空会话；焦点始终可见。

### 任务 7：完整验证

步骤：

1. 运行 `yarn test`。
2. 运行 `yarn build`。
3. 检查控制台错误和警告。
4. 在 390×844、768×1024、1024×720、1280×720 四个视口验证：
   - 空状态。
   - 长回答。
   - 至少 20 条引用。
   - 两轮对话及历史证据切换。
   - 服务错误、证据不足和用户取消。
5. 记录实际 `scrollHeight/clientHeight/scrollWidth`，确认根页面无意外滚动和横向溢出。
6. 检查最终 diff，确保没有修改登录或无关后端功能。

## 5. 完成标准

- 所有新增行为都有自动化测试或明确的浏览器验证证据。
- `yarn test` 与 `yarn build` 都以退出码 0 完成。
- 关键视口均完成实际页面验证，而不是只检查 Tailwind 类名。
- 页面不再暴露无必要的内部路径和原始检索 JSON。
- 不宣称远端部署完成，不自动提交、推送或创建 PR；完成后由用户选择集成方式。
