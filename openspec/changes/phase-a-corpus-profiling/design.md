## Context

知识库已有 parsed JSON、cleaned Markdown、chunk JSONL、质量检查、报告策略、制品清单和阶段验收框架，但语料质量仍以结构存在性检查为主。现行路线图要求先建立 profiling，再逐步把稳定指标提升为门禁。项目必须保持离线确定性流水线，不允许外部模型修改主知识数据或成为默认运行依赖。

阶段 A 同时面对两类信号：字符、结构和引用关系可以确定性计算；OCR 语义质量更适合由模型辅助判断。两类结果必须隔离，避免网络、密钥、模型版本和输出漂移影响确定性发布。

## Goals / Non-Goals

**Goals:**

- 对 parsed、cleaned、chunks 的 `doc_id` 并集生成稳定文档级画像和中文汇总报告。
- 对空正文、替换字符、重复 `doc_id` 和孤儿 chunk 建立确定性阻断。
- 通过 YAML 管理排除、阈值、抽样与 Provider 策略。
- 提供通用 OCR 评估 Provider 契约、mock 和 DeepSeek 首个真实适配器。
- 把确定性画像接入主流水线、质量检查、制品登记与阶段验收。

**Non-Goals:**

- 不替换现有解析器，不执行 OCR 或版面重建。
- 不上传完整 cleaned 文档。
- 不让模型输出直接阻断流水线、修改语料或调整阈值。
- 不实现 topic 分类、层级 chunk、结构化候选或增量构建。

## Decisions

### 1. 使用双轨数据集

确定性数据写入 `corpus_profile.jsonl`，模型数据写入 `corpus_ocr_assessments.jsonl`。中文报告读取两者并明确标注模型状态。相比把模型字段混入画像，这一方案保证基础数据可复现；代价是报告生成需要合并两个输入。

### 2. 以三层并集为画像边界

加载 parsed 文件中的 `doc_id`、cleaned 路径推导的逻辑文档标识以及所有 chunk 的 `doc_id`，取并集后逐项记录阶段存在性。README 由配置排除，seed/context 保留。相比只统计完整三元组，并集能暴露孤立和缺失阶段。

### 3. 问题分为阻断与告警

阻断代码仅覆盖 `empty_cleaned_content`、`replacement_character`、`duplicate_doc_id` 和 `orphan_chunk_document`。长度、表格、异常符号、空标题、重复标题及阶段缺失为告警。相比把启发式指标直接设为错误，这一方案降低误报对流水线的影响。

### 4. 画像先落盘再返回失败

即使发现阻断问题，命令也必须原子写出完整数据集和报告，再返回非零退出码。这样 CI 能阻断，同时操作者仍能获得诊断证据。若输入无法解析到足以生成可信画像，则保留旧输出并立即失败。

### 5. 模型使用通用接口和严格结构化边界

Provider 接口接收文档标识、输入指纹和固定抽样，返回 `ok`、供应商、模型和结构化内容。mock 用于离线测试；DeepSeek 适配器复用现有 HTTP 客户端的认证、超时和错误归一方式。模型只能给出 `low|medium|high` 风险、理由和人工建议，治理字段由系统补齐。

### 6. 固定抽样与预算优先于并发吞吐

每篇取首、中、尾片段并去重，配置最大文档数、单文档输入字符、总输入字符和并发上限。首版实现可保持串行，但必须拒绝超过配置上限的调度；后续增加并发不改变契约。模型失败项结构化记录为 `skipped` 或 `failed`，不覆盖已有成功记录。

### 7. 主流水线保持离线

主流水线运行确定性画像，并让 OCR 评估以默认 disabled/skip 模式刷新状态，不访问网络。真实 DeepSeek 调用只能通过显式 CLI 参数启用。质量检查只验证两类数据集的结构和确定性阻断结果，不把模型风险作为错误。

## Risks / Trade-offs

- [cleaned 文件名到 `doc_id` 的推导可能与 parsed 标识不一致] → 优先读取文档内可用元数据，并用来源相对路径与 chunk 关联测试固定规则。
- [启发式表格或异常符号误报] → 仅作为告警，阈值全部配置化。
- [模型上传文本存在边界风险] → 只发送固定抽样，记录输入指纹而不保存原始响应，密钥只从环境变量读取。
- [模型结果漂移] → 独立数据集保存 provider、model、prompt_version、input_fingerprint 和状态，不参与确定性门禁。
- [当前语料暴露历史硬错误] → 画像先产出诊断；若存在问题，修复语料或经明确设计调整规则，不静默降级门禁。

## Migration Plan

1. 先加入配置、Schema 和失败测试，不改变现有流水线。
2. 实现确定性画像并在当前语料上生成基线。
3. 实现可选 OCR Provider、mock 和 DeepSeek 跳过/失败边界。
4. 注册报告、制品 producer、质量检查和主流水线步骤。
5. 新增阶段 A gate，运行目标测试、全量测试和阶段验收。
6. 回滚时移除新增步骤与配置即可；新增产物均为可再生成派生文件，不影响主知识数据。
