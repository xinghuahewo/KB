## 背景

项目已有稳定 URI、JSON-LD context、实体/来源/chunk 目录和人工复核流水线，但阶段五标准出口尚未实现。标准出口必须可复跑、可审计并保持现有主发布格式稳定；模型仅作为语义候选助手。

## 目标与非目标

**目标：**

- 生成 JSON-LD 实体与来源目录、PROV-O 溯源记录和 Turtle 样例。
- 生成模型辅助映射候选，并用人工审核和显式写入控制其生效。
- 将阶段五纳入流水线、制品清单和验收门禁。

**非目标：**

- 不迁移主存储到 RDF 或图数据库。
- 不引入 OWL 推理。
- 不允许模型修改主实体、主关系、复核状态或直接生成正式出口。
- 不要求默认流水线联网或提供模型密钥。

## 技术决策

1. **正式出口完全确定性。** 使用 YAML 映射配置和已批准映射生成排序稳定的 JSON-LD、JSONL 与 Turtle。相比模型直接导出，该方案可复跑、可测试、可审计。
2. **复用阶段三点五 URI。** `semantic_identity.yaml` 是唯一 URI 来源，避免出现两套不兼容标识。
3. **未知关系保留项目词汇。** 已知关系优先映射 SKOS/PROV/Schema；无法安全映射时使用 `bgpkb:` 谓词并进入候选队列，不强行套用外部词汇。
4. **模型候选与正式发布隔离。** mock 为默认 provider，DeepSeek 必须显式启用。候选固定为 `pending_review`，结构化校验失败即丢弃。候选 ID 包含来源证据、输入内容和 prompt version 的指纹，输入变化会使旧批准失效。
5. **人工批准产生覆盖层。** 人工 CSV 经审计后，由 `--write` 生成稳定的批准映射 JSONL；approved 必须具有审核人、合法审核时间和当前输入指纹。同一本地项出现多个不同批准映射时整组阻塞。
6. **无新增强制依赖。** 使用 Python 标准库和已有 PyYAML；Turtle v1 仅支持 IRI 主语、CURIE 谓词、IRI/CURIE 对象和纯字符串 literal，其他输入显式拒绝。

## 风险与权衡

- 模型可能给出错误映射 → 仅生成候选，必须有来源证据并经人工批准。
- 手写 Turtle 转义范围有限 → 明确限制 v1 数据类型，用测试覆盖全部允许类型和拒绝路径。
- pending 实体进入标准目录可能被误解 → 保留 `review_status`，报告明确区分治理状态。
- 人工批准覆盖层可能陈旧 → 候选 ID 包含输入指纹，审计时必须与当前候选版本一致。
- 阶段五增加流水线耗时 → 默认 mock 与确定性步骤均为本地文件处理，不调用网络。

## 迁移方案

1. 先新增配置、Schema 和独立生成脚本，不接入主流水线。
2. 生成并验证标准出口，不修改现有发布文件。
3. 接入模型候选和人工审核，默认保持无批准映射。
4. 验证通过后接入主流水线与阶段 gate。
5. 回滚时删除新增步骤和派生出口即可，现有 JSONL、CSV、SQLite 不受影响。
