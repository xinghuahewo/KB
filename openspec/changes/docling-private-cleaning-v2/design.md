## Context

现有 `parse_documents.py` 负责 TXT、HTML、YAML 和 PDF 文本抽取，cleaned Markdown 与 chunks 是主要下游输入。阶段 A 已建立文档级画像和四类确定性门禁，但解析层仍缺少统一版面结构、阅读顺序、表格单元格、代码、公式、图片引用、bbox、OCR 证据和逐次清洗审计。

本变更运行于独立 Linux GPU 服务器。用户选择 Docling 单主引擎、现有解析器显式 fallback、结构化 Block JSON 权威数据、自适应 OCR、全部 54 篇并行 v2 迁移、保真优先和生产运行离线。实时 API 与图片语义解释已明确排除。

## Goals / Non-Goals

**Goals:**

- 建立锁定镜像、模型 hash 和禁止外网的 Docling 私有运行环境。
- 建立与 Docling 内部对象解耦的 Canonical Block JSON v2。
- 建立可追溯的分级清洗、转换审计和人工审核隔离。
- 建立幂等、可恢复、可隔离单篇失败的全量批处理。
- 对全部语料生成并行 v2，达到结构、OCR 和差异门槛后原子切换。

**Non-Goals:**

- 不提供实时或批处理 HTTP API。
- 不引入 PP-StructureV3、Unstructured 或第二个版面引擎。
- 不生成图片/图表语义解释。
- 不自动重写正文事实，不重做分类、实体或关系抽取。

## Decisions

### 1. 生产运行使用离线 GPU 容器

构建阶段预下载并锁定依赖与模型，生产容器禁止外网。镜像摘要、模型 hash、CUDA/驱动、Docling 和配置版本写入 run 记录。相比运行时自动下载，这能避免供应链、版本漂移和不可复现问题；代价是升级必须经过显式构建与验收。

### 2. Docling 主引擎与显式 fallback

所有支持格式默认由 Docling 解析。现有解析器只在明确配置和 Docling 失败时执行，结果必须标记 fallback、原因和引擎版本，未经审核不能发布。相比彻底移除旧解析器，这提供迁移期安全网；相比静默 fallback，它保留质量边界。

### 3. 项目 Canonical Block 隔离 Docling 变化

适配层把 Docling 输出转换为项目自有 Block Schema。权威 cleaned Block 保存结构、bbox、阅读顺序、特殊对象、质量、追溯和治理字段；Markdown 与 chunks 只做派生。这样下游不绑定 Docling 私有对象，升级时只需改适配器。

### 4. 自适应而非强制整页 OCR

预检按文本层覆盖、图片比例、字符量和异常信号选择 OCR。原生文本和 OCR 证据并行保存，清洗层记录最终选择。相比所有页面 OCR，这减少误识别和资源消耗；相比纯人工触发，它能覆盖规模化迁移。

### 5. 特殊内容结构化保留

表格保存行列、表头、合并单元格和来源页；代码、公式和图片保存类型化表示、bbox 与相邻标题。图片资产可以裁剪保存，但不产生语义解释，避免模型生成内容进入清洗事实层。

### 6. 清洗规则分级并保存 transformation

无损规范化可以自动执行；结构校正必须保存规则、前后值、证据和置信度；语义性修改不能自动进入正式 Block。所有删除、替换、合并和拆分都必须有 transformation，禁止静默清洗。

### 7. 文档级幂等状态机

输入 hash、镜像、模型与配置共同形成处理指纹。相同指纹复用成功结果；状态机支持断点续跑。暂态错误有限重试，内容和 Schema 错误直接隔离。单篇失败不终止批次，输出先写临时目录再原子发布。

### 8. 并行 v2 与发布指针切换

v2 产物独立存放，不覆盖 v1。全部文档完成差异、人工高风险验收和硬门禁后更新发布指针；v1 目录、manifest 和回滚命令保留。相比 Git 快照后原地覆盖，这能隔离迁移风险并支持即时回滚。

### 9. 保真优先的验收

全量门禁覆盖文档完整性、Schema、追溯、稳定性、表格、fallback 和正文覆盖。约 12 篇高风险文档用于标题层级、阅读顺序、表格和 OCR 人工评测。性能数据用于容量基线，但不会以牺牲结构保真换吞吐。

## Risks / Trade-offs

- [Docling 或模型升级改变输出] → 固定镜像与模型 hash，适配器 fixture 和全量差异回归。
- [GPU OOM 或超长文档拖垮批次] → 文档级资源预算、有限重试、隔离和断点续跑。
- [OCR 覆盖正确原生文本] → 并行保存原生/OCR 证据，自适应选择并审计。
- [结构清洗误删正文] → transformation、正文覆盖门槛和逐文档差异。
- [单引擎存在盲区] → 现有解析器显式 fallback；fallback 未审核不发布。
- [v2 迁移影响既有下游] → 并行目录、发布指针、全链回归和 v1 回滚。
- [模型镜像体积和许可证复杂] → 生成 SBOM、许可证清单和模型 manifest。

## Migration Plan

1. 建立容器、依赖/模型锁和离线启动验证。
2. 建立 Canonical Schema、Docling fixture 与适配器，不接入生产指针。
3. 建立预检、自适应 OCR、清洗规则和转换审计。
4. 建立批处理状态机、重试、隔离、断点续跑和报告。
5. 全量生成 parsed v2、cleaned Block v2、Markdown v2、assets v2 和 chunks v2。
6. 运行 54 篇差异、稳定性、人工高风险集和下游回归。
7. 通过阶段验收后原子切换发布指针；观察期内保留 v1。
8. 回滚时恢复 v1 指针并重新生成发布 manifest，不删除 v2 诊断证据。

## Open Questions

- 实施前需确认目标 GPU 型号、显存、驱动与 CUDA 基线，以选择兼容且可锁定的 Docling/模型版本；该选择不改变本规格的数据和治理契约。
