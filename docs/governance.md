# 工程治理

## 分支与合并

- `main` 禁止直接推送；功能通过受保护分支和 PR 合并。
- PR 必须通过无制品后端测试、API 契约、前端测试/构建、仓库卫生和 OpenSpec 校验。
- 真实制品门禁在受控服务器或发布流程运行，不向普通 CI 下载 2 GB 制品。
- 未通过门禁不得合并、部署或移动 current 指针。

2026-07-13 已通过 GitHub API 验证 `main`：必需检查为严格模式 `verify`，管理员同样受约束，PR 对话必须解决，并禁止强推和删除分支。PR #1 已实际触发必需检查和未解决对话拦截，不能绕过保护直接合并。

## Git 数据边界

普通 Git 禁止跟踪 SQLite、数据库、Parquet、FAISS、NumPy 向量、索引、大型语料和生成报告。`.gitignore` 只是第一道边界；CI 还检查被强制加入的忽略文件和超过 1 MiB 的 blob。

大型数据应进入服务器制品库、对象存储、发布制品或确有跨仓库版本协作需求时的 Git LFS。当前项目的运行制品默认使用服务器制品库，不使用普通 Git 历史。

旧分支退役前必须先生成可校验的 Git bundle，并在[历史文档映射](archive-map.md)记录分支提交、制品路径和 SHA-256；确认服务器副本一致后才可删除远端引用。

## 变更与测试

- 行为变更先写失败测试，再做最小实现。
- 默认测试必须能在无制品干净克隆中运行。
- 依赖真实语料、SQLite 或向量索引的测试必须显式标记 `artifact`。
- API 路由和方法由无制品 OpenAPI 契约测试锁定。
- 完成声明前必须重新运行测试、构建、卫生检查和规格校验。

## 文档规则

- 文档始终使用中文。
- 长期事实只在一个权威文档维护；其他位置使用链接。
- 关键取舍写 ADR，发布变化写 CHANGELOG，执行中需求写 OpenSpec。
- 阶段计划完成后归档 OpenSpec，并删除重复阶段文档。
- 密钥、密码、私钥、token 和个人凭据不得进入文档。

## 发布记录

每次正式发布必须记录代码提交、前端构建标识、制品 release id、`SHA256SUMS` 哈希、部署时间、验收结果和回滚点。
